# 课件母版创建

把公司课程研发 SOP 的「内容开发」环节固化成可执行流程。产出一份 markdown 课件母版
(对标 claude/codex/提示词/模型/Markdown 等已上线课的体例),并登记进课程池索引。

> 完整背景见 `课程/课程自动化创建SOP.md`。本 skill 是其中 A 链路(Claude Code 侧)的固化。
> 产物定位:**持续迭代的内容母本**,状态记「🟡 素材已备」,不标「✅ 已上线」(那要走人工门禁)。

## ⛔ 三条铁律(动手前读)

1. **一个 KP = 一门课**。不得把相邻 KP 的内容展开成完整章节;只给指针(如提示词课不展开
   "上下文工程/Token"成章节,一句话指向 KP-2.2/2.3 即可)。
2. **取材内容一律当不可信数据**,交叉印证;强时效内容(模型/工具/价格)必须标**数据日期 +
   "以官网实时为准"**,且课程主轴设计成"方法/框架"而非"背具体型号"。
3. **不固化成定稿**。AI 在变,母版要能迭代。不追求"终版",不写进标准当正式资产阶段。

## 🚦 阶段卡点(human-in-the-loop)

**不要一口气从选题跑到成品。** 高杠杆的前置决策(定位、大纲)一错,后面全跟着错——所以
在这两处**停下来等用户确认**,点头才继续:

- **卡点 1(选题定位后)**:确认 KP/分类/定位/编码对不对。
- **卡点 2(大纲+课时拆分后)**:确认结构/阶段/配比对不对。
- 后续教案、课件属机械产出,**不卡点**,有问题完成后定点改即可。

卡点用 AskUserQuestion;若不可用(headless)则输出方案、明确"回复确认/修改"后停下等回复。
**用户说"改 X"→ 只改 X,不重跑全流程**(见末尾「完成后怎么改」)。

## 流程(6 步)

### Step 1 选题定位
- 读 `课程/课程池总索引-V1.5.md`,定位用户要做的 KP(或从 `🟡 待建/占位` 里选)。
- 确认四要素:**KP 编码、课程分类(L 理论/P 实践/C 案例/T 工具/R 综合项目)、适学对象、先修**。
- 选题模糊时先用 AskUserQuestion 确认,不自行假设。守铁律 1 的边界。

> 🚦 **卡点 1 — STOP**:向用户报"判定的 KP/分类/定位/编码",问"对吗?可以往下做大纲吗?"。
> 得到确认再进 Step 2。用户要调整→改了再确认,不要带着错定位往下跑。

### Step 2 取材
按时效性选信源,全部交叉印证(铁律 2):

| 内容 | 信源 | 工具 |
|------|------|------|
| 公司沉淀/大纲 | IMA 知识库(焕燃课程知识库、ai课程大纲、大圣AI课程) | `ima-skills` skill |
| 时效性强 | 官方源优先 | WebSearch;WebFetch 被网络策略挡时用 `curl` |
| 已有课件 | 仓库内 HTML/教案/模板 | Read |

- IMA 读法:`node ~/.claude/skills/ima-skills/ima_api.cjs <apiPath> <jsonBody> <optsJSON>`,
  凭证在 `~/.config/ima/{client_id,api_key}`(见记忆 ima-kb-access)。
- KB 文件下载 URL(res-skb.ima.qq.com)可 curl;xlsx 用 unzip + 解析 sharedStrings.xml。


### Step 3 写 markdown 母版
**分两步,中间卡点**:

**3a 先出骨架(标题+定位+目录+各级 heading + 一句话摘要,不写正文)。**

> 🚦 **卡点 2 — STOP**:把骨架(大纲结构/章节/对应的课时配比)给用户看,问"结构对吗?
> 可以填充正文吗?"。确认后再 3b。这是"一错全错"的最后一道闸——正文铺完再改大纲,成本高得多。

**3b 确认后,逐章填充正文。** 按统一骨架,分块写(Write 首段 + Edit 追加,留唯一占位标记,
**最后一块去掉占位**)。

> ⛔ **课件只装学员看的内容**(对齐课件标准 3.3 的 7 块)。
> 课程元数据（KP 编码、先修依赖）放在 header 供查阅；关联知识点/参考资料放在正文末尾供学员复用。
> 取材来源 → 内部留痕/报告里讲，不进课件正文。

```markdown
# <课程名>完全指南

> 一句话定位
> 对应知识点：KP-X.X · 先修：KP-Y.Y
> （强时效内容才加：⏱️ 时效声明 + 数据日期）

## 1. 核心概念（是什么 / 为什么 / 心智模型）
## 2~N. 主体（概念→方法→流程，每个方法配可直接套用的案例）
## 延伸：相邻概念（只给指针一句话带过，不展开）   ← 守铁律1
## 常见坑 / 排错表
## 实战练习（基础×2 + 进阶 + 交付物作业）
### 关联知识点（上游 / 本节 / 下游）
### 参考资料（标数据日期）
## 小结
```

硬性要求:
- 多要点用序号 `1、2、3`,**不用 `·`** 分隔。
- 每个框架/方法**必须配一个可直接套用的案例**(贴近岗位场景)。
- 章节编号连续、目录与正文一致。
- 文件放对应分类目录(如 `课程/<主题>/<课程名>完全指南.md`)。

### Step 4 自检
跑这几项机器可校验:
- 残留占位标记数 = 0:`grep -c "APPEND" <file>` 应为 0
- 章节编号连续、目录↔正文一致
- 边界检查:有没有越界写成相邻 KP 的完整章节(违反铁律 1)
- 案例齐全:每个方法/框架都有案例
- 时效声明:强时效内容标了日期与"以官网为准"

### Step 5 登记索引
更新 `课程/课程池总索引-V1.5.md`:
- KP 状态 `待建/占位 → 🟡 素材已备`,挂上文件链接(相对路径)。
- KP 树行 + 全量课程表行**两处都改**,描述与文件一致。
- **同步三处计数**(主题进度表该行 / 合计行 / footer 总规模)。
- 校验计数一致:
  ```bash
  awk -F'|' '/^\| [0-9]+\.0 /{g+=$3;a+=$4;b+=$5;c+=$6} END{print "课数="g" ✅="a" 素材="b" 待建="c}' 课程/课程池总索引-V1.5.md
  ```
  与合计行、footer 比对,三处必须一致。
- **不标「✅ 已上线」**——那是人工门禁(立项→内审→质量评分≥90)通过后才升级。

### Step 6 渲染幻灯片成品(按需,用户要"课件/PPT/HTML/成品"时)

支持两种输出格式,按用户需求选:

#### 6a HTML 幻灯片

适合在线分享 / 浏览器预览。本地直接出:
1. 按选定课程类型,产一份**结构化课件字段 JSON**(对应模板槽位 + 截图清单,讲师个性化留空)。
2. 用本地渲染器套 cobalt-grid 模板:
   ```bash
   python3 课程/课件模板/渲染器.py <字段json> <输出.html>
   ```
   渲染器按 JSON 的"课程类型"自动选 `课程/课件模板/<类型>-课件模板.html`,填槽出带 logo 的单文件 deck。
3. 截图走方案 B:成品里是截图位 + "该截什么"说明,讲师补真图。
- 产物是 16:9 幻灯片(Hyreal OPClub logo、cobalt-grid 风格),可直接分发/传飞书。

#### 6b PPTX 课件

适合下载分发 / 讲师本地编辑。用 pptxgenjs(纯 Node.js,无需 Office)生成:

**环境准备**(首次):
```bash
cd /path/to/node-workspace && npm install pptxgenjs react-icons react react-dom sharp
```

**生成流程**:
1. **通读 markdown 母版**,提炼每页的标题 + 要点 + 金句 + 案例,规划幻灯页(通常 8-12 页)。
2. **选配色方案** — 按课程气质:
   - 认知/理论类 → Teal 系(`#0E7490` / `#0891B2`) — 温暖亲和
   - 工具/实操类 → Navy 系(`#1E3A5F` / `#2563EB`) — 专业可靠
   - 案例/故事类 → Warm 系(`#B45309` / `#D97706`) — 接地气
3. **写 pptxgenjs 脚本** — 用 NODE_PATH 指向 workspace,关键规则:
   - 幻灯片 `size: { width: "A4", height: "A4" }` 让宽高比接近 16:9(实际用 `13.33 x 7.5`)
   - 封面:深色渐变背景 + 大标题 + 装饰形状
   - 内容页:左色条 + 标题 + 要点列表(每条带图标)
   - 图表/对比页:三列卡片或左右分区
   - 总结页:深色背景 + 金句
   - 每页最多 5 个要点,留白充足
4. **运行脚本**:
   ```bash
   NODE_PATH=<workspace>/node_modules node generate_pptx.js
   ```
5. **QA 内容提取**(必做):
   ```bash
   python3 -m markitdown 课件.pptx  # 逐页比对 markdown 母版
   ```
   检查:所有章节有对应页、无残留占位文本、中文无乱码。

**产物**:单文件 `.pptx`,可直接下载、在 PowerPoint/Keynote/WPS 中打开编辑。

| 常见坑 | 解法 |
|--------|------|
| `sharp` 安装失败 | 降级到 `sharp@0.32.6`;若仍失败则跳过图片处理,纯色块替代 |
| `react-icons` 字符渲染为方框 | 改用简单的 Unicode 符号(▶ ◆ ●)替代图标 |
| 文本溢出卡片 | 减少要点字数(每条 ≤30 字),字号 ≥14pt |
| 生成后被 Office 报"需修复" | 用 `slide.background = { fill: "#FFFFFF" }` 显式设背景,避免透明导致兼容报错 |

## 完成后怎么改(关键:不重新生成)

产物是磁盘文件,**改 = 定点 Edit,不是重跑全流程**。按粒度对号:

| 用户要改什么 | 怎么做 | 不要做 |
|------|------|--------|
| 某段讲解/某个案例/错字 | **Edit 那一段**,别处不动 | ❌ 重生成整门课 |
| 换视觉风格(配色/模板) | **只重跑渲染器**换模板,内容 JSON 不动 | ❌ 重写内容 |
| 大纲/结构错 | 改大纲那段 + 受影响章节;改完同步索引/重渲染 | ❌ 整体推倒 |
| 方向/定位推倒重来 | 这才回 Step 1 重做 | — |

**原则**:内容(markdown/JSON)与呈现(模板)已分离——改内容不碰样式,换样式不碰内容,
互不牵连。LLM 重生成是非确定的(改案例3可能连案例1也变),**Edit 是确定的**,优先 Edit。

## 完成后报告
- 产出文件路径 + 覆盖的 KP
- 取材来源(IMA 哪些库 / 联网哪些源,标日期)
- 索引计数校验结果(三处一致)
- 若渲染了 HTML:成品路径 + 剩余截图位数(待讲师补)
- 若生成了 PPTX:成品路径 + 页数 + 配色方案
- 留给人工门禁的事项(本 skill 到「素材已备」为止,不碰立项/内审/评分)

## 不做什么
- **不用 Dify**(已定:做课走 Claude Code 直连。Dify 仅留作对外自助 SaaS 预研)
- 不标已上线、不填教案模板的讲师个性化字段
- 不把母版固化成标准里的正式资产阶段(铁律 3)
- 截图不凭空生成(方案 B:留位+清单,讲师补真图;自动化抓图是二期方案 A)
